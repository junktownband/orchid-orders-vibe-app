# Master Context для Codex

## Задача

Orchid Control — локальное web/PWA-приложение для учета ремонтной мастерской. Приложение должно заменить Excel/Google-таблицу, но не повторять ее интерфейс. Таблица была прототипом домена: заказы, клиенты, инструменты, услуги, мастера, расходы, налоги, комиссии и выплаты.

Источник истины — локальная PostgreSQL база через Prisma. Google Sheets, Forms, Apps Script, Drive API и любые Google-интеграции не используются.

## Пользовательская идея

Главный UX-принцип:

> Открываешь приложение, видишь заказы и деньги, быстро создаешь заказ или расход, а расчеты выполняются автоматически.

Интерфейс должен быть простым для сотрудников, которые не хотят разбираться в тяжелых CRM/ERP.

## Домен

- ремонтные заказы;
- клиенты;
- инструменты;
- услуги и расходные материалы;
- ответственные мастера по услугам;
- статусы ремонта;
- оплаты и способы оплат;
- обычные расходы;
- системные расходы по налогам и зарплатам;
- категории расходов;
- комиссии мастеров;
- выплаты комиссий;
- аналитика по выручке, расходам, невыплаченным комиссиям и чистым деньгам.

Дебиторка сейчас намеренно не является частью dashboard-контракта: частичные оплаты еще не оформлены как полноценный бизнес-процесс, поэтому этот показатель создает шум.

## Роли

- `OWNER` — владелец, видит все и управляет настройками, пользователями, финансами и выплатами.
- `ADMIN` — управляет операционными разделами, кроме критичных owner-only настроек.
- `MANAGER` — создает и ведет заказы, расходы и оплаты, видит операционную аналитику.
- `MASTER` — работает только с заказами: видит все заказы и может переводить ремонт в рабочие статусы `IN_PROGRESS`/`READY`. Бэкофис, деньги, расходы, настройки, каталог услуг, комиссии и аналитика ему не показываются.
## Финансовые правила

1. За основу берется текущий итог заказа, рассчитанный из позиций. При выдаче его не вводят вручную.
2. Если включена самозанятость, при выдаче заказа выбирается тип клиента для налога: физлицо или юрлицо.
3. Налог считается от текущего итога заказа до комиссий; если расходник услуги уже добавлен клиенту в итог заказа, он входит в налоговую базу.
4. Налог фиксируется системным расходом `TAX`; вручную такой расход создать нельзя.
5. Обычные расходы можно добавлять на организацию, заказ или конкретную услугу заказа.
6. Комиссия мастера считается по каждой услуге отдельно, после налога и после расходников этой услуги.
7. Выплата комиссии мастеру фиксируется через раздел выплат и создает системный расход `SALARY`; вручную зарплатный расход создать нельзя.
8. Чистые деньги на dashboard считаются как оплаченная выручка минус подтвержденные расходы минус невыплаченные комиссии.

## Визуальный стиль

Приложение mobile-first: темный фон, аккуратные панели, крупные читаемые цифры, iOS-like bottom navigation, яркие акценты только для важных действий и статусов. Desktop — расширенная версия мобильного интерфейса, а не отдельная табличная CRM.

## Принцип разработки

Каждый этап должен:

1. прочитать актуальные документы в `docs/`;
2. внести законченный набор изменений;
3. добавить или обновить тесты для бизнес-логики;
4. проверить типизацию, тесты и сборку;
5. обновить `docs/CHANGELOG.md`, `docs/ROADMAP.md` или профильные документы, если менялись правила.
