# Stage 13 — Operations Settings, Tax, Audit

## Цель

Собрать следующий продуктовый этап без поломки текущих сценариев: расходы должны удобнее связываться с заказами, настройки должны покрывать самозанятость и управление мастерами, аналитика должна учитывать timezone организации, а важные действия должны попадать в audit log и отображаться в интерфейсе.

## Текущее состояние

- `Expense.repairOrderId` уже nullable, поэтому общий расход можно делать без разрушительной миграции.
- `Expense.repairOrderItemId` уже есть, поэтому расход можно относить к конкретной позиции заказа.
- Форма расхода больше не должна быть основным местом привязки к заказу: из карточки заказа контекст уже известен.
- Поиск заказов уже есть на backend и поддерживает поиск по `orderNumber`, клиенту, телефону, инструменту и мастеру.
- `RepairOrder.orderNumber` хранится строкой; старый формат был `R-00001`, новый формат должен быть `00001`.
- `OrganizationSetting.taxMode` добавлен для режима самозанятости.
- `Membership.commissionPercent` уже есть; settings API и UI управления мастерами добавлены в `Настройки -> Мастера`.
- `RepairOrderItem.assignedMasterMembershipId` и commission snapshot поля добавлены, чтобы комиссия считалась на уровне услуги, а не заказа.
- `AuditLog` и `AuditAction` уже есть в Prisma schema. Важные события заказов, расходов и каталога пишутся в общий журнал, доступный через API и `Настройки -> Журнал`.
- Аналитика использует timezone организации для локальных границ месяца и переводит их в UTC перед запросом к БД.

## Продуктовые решения

### Расходы

- Страница расходов — реестр и место для общих расходов.
- Форма нового расхода без контекста заказа начинает с поиска заказа по номеру/клиенту/телефону/инструменту/мастеру, без dropdown списка заказов.
- Общий расход создается только через явную галочку `Общий расход`; по умолчанию галочка выключена.
- Расходы, связанные с заказом, также создаются из карточки заказа.
- В карточке заказа пользователь выбирает:
  - расход к заказу целиком;
  - расход к конкретной позиции заказа.
- Расходы к позиции участвуют в чистой базе комиссии этой услуги после подтверждения.
- Расходы к заказу целиком не уменьшают базу конкретного мастера, пока не утверждено правило распределения.
- `TAX` — системный вид расхода; руками через `POST /expenses` создать нельзя.

### Идентификатор заказа

- Не менять имя поля `orderNumber` в API.
- Новые заказы хранить и показывать пользователю как `№ 00001`, без префикса `R-`.
- Backend должен принимать в поиске оба формата: `R-00001` и `00001`.
- Dev migration нормализует старые значения `R-00001` в `00001`.

### Самозанятость

- В настройках организации добавить режим налогообложения:
  - `Без налогового учета в Orchid`;
  - `Самозанятость`.
- Для самозанятости ставки фиксируются как справочные значения:
  - `4%` — доходы от физических лиц;
  - `6%` — доходы от юридических лиц и ИП.
- При выдаче заказа, если включена самозанятость, пользователь выбирает radio:
  - `Физическое лицо`;
  - `Юрлицо или ИП`.
- При выдаче заказа сохранять snapshot налогового режима, типа плательщика, ставки и суммы налога.
- При выдаче заказа автоматически создавать confirmed `TAX` expense на сумму налога.
- Интеграцию с `Мой налог` не делать сейчас; добавить только distant roadmap item.

### Пользователи и мастера

- Owner/Admin видят экран `Настройки -> Мастера`.
- Admin может добавлять мастеров, редактировать карточку мастера и менять `commissionPercent`.
- Admin может отключать/удалять мастеров через soft deactivate membership, не физически удаляя пользователя из истории.
- Admin не может удалять, отключать или понижать другого Admin/Owner.
- Owner может управлять шире, но удаление Owner запрещено.
- В истории заказов сохранять старые связи с мастером; отключенный мастер просто не доступен для новых назначений.

### Комиссия мастера

- Мастер назначается на `SERVICE`-позицию заказа.
- Старое поле `RepairOrder.assignedMasterMembershipId` остается как общий ответственный/координатор и fallback для старых данных.
- При выдаче заказа комиссия фиксируется snapshot-полями на строке услуги.
- Выплата мастеру фиксируется отдельным статусом `UNPAID/PAID` на строке услуги.
- Owner/Admin видят реестр комиссий и могут отметить выплату мастеру.
- Итоги реестра комиссий считаются по всем начислениям организации, даже если UI показывает только последние строки.
- Выплата комиссии не создает обычный расход, чтобы не было двойного вычитания.
- Формула базы:
  `allocatedServiceRevenue - allocatedTaxCents - item.costCents - confirmedRegularItemExpenses`.
- Позднее подтверждение обычного расхода по конкретной услуге пересчитывает unpaid-комиссию выданного заказа; оплаченные строки не переписываются.
- `allocatedServiceRevenue` распределяет текущий итог заказа пропорционально ценам строк.
- `allocatedTaxCents` распределяет налог заказа пропорционально выручке строк.
- Комиссия считается только по `SERVICE`; материалы, товары, струны и прочие resale-позиции не дают комиссию мастеру.
- Налоговый `TAX` расход уменьшает базу комиссии мастера только своей долей по строке услуги.

### Audit

Писать audit log для:

- изменения статуса заказа — реализовано;
- отметки оплаты — реализовано;
- выдачи заказа — реализовано;
- подтверждения расхода — реализовано;
- создания позиции каталога — реализовано;
- отметки выплаты комиссии мастеру — реализовано;
- изменения/отключения позиции каталога — реализовано;
- создания/изменения/отключения мастера — реализовано;
- изменения комиссии мастера — реализовано через audit записи `Membership`;
- изменения налоговых настроек — реализовано.

UI:

- общий журнал в `Настройки -> Журнал` — реализовано;
- timeline на карточке заказа — реализовано.

### Customer Edit

- Модель истории клиента зафиксирована: `Customer` — канонический изменяемый профиль, изменения пишутся в `AuditLog`, order-level snapshots пока не вводятся.
- Редактирование клиента доступно из карточки заказа.

## План реализации

### 1. Расходы из карточки заказа

Backend:

- Сохранить существующий `POST /api/v1/expenses` contract: `repairOrderId` и `repairOrderItemId` остаются optional.
- Явно запретить ручной `kind = TAX` в create schema.
- Проверять, что `repairOrderItemId` принадлежит выбранному заказу.

Frontend:

- В `OrderDetailPage` добавить действия:
  - `Расход к заказу`;
  - `Расход к позиции`.
- В `ExpenseCreatePage` читать `orderId`/`itemId` из URL query и подставлять контекст.
- Без `orderId` форма показывает поиск заказа; checkbox `Общий расход` по умолчанию выключен.
- Страницу расходов оставить реестром с подтверждением черновиков.

Tests:

- expense create from order card sends linked `repairOrderId`;
- expense create from item action sends linked `repairOrderItemId`;
- general expense sends no `repairOrderId`;
- manual `kind = TAX` is rejected.

### 2. Удобный номер заказа

Backend:

- Ввести helper `formatOrderNumber(sequence)` и `normalizeOrderNumberQuery(query)`.
- Новые заказы создавать как `00001`.
- Нормализовать существующие dev-данные миграцией из `R-00001` в `00001`.
- Search должен поддерживать оба формата.
- Решение о миграции старых номеров принять отдельно после проверки данных.

Frontend:

- Показывать `№ 00001` вместо `R-00001`.
- В поиске и подсказках не требовать префикс.

Docs:

- Зафиксировать правило: пользователю виден номер без технического префикса.

### 3. Настройки самозанятости и выдача заказа

DB migration:

- Добавить enum или string поле налогового режима в `OrganizationSetting`.
- Добавить в `RepairOrder` поля snapshot:
  - `taxModeSnapshot`;
  - `taxSubject`;
  - `taxRateBps`;
  - `taxAmountCents`.

Shared/API:

- Расширить settings schemas.
- Расширить `issueRepairOrderSchema` полем `taxSubject`.
- Если tax mode включен, требовать `taxSubject` при выдаче заказа.

Backend:

- Settings module: `GET /settings`, `PATCH /settings/tax`.
- Issue flow считает налог от текущего итога заказа, рассчитанного из позиций:
  - individual: `round(order.totalAmountCents * 400 / 10000)`;
  - legal/IP: `round(order.totalAmountCents * 600 / 10000)`.
- Сохранять snapshot, чтобы изменение настроек не переписало старые заказы.
- Создавать confirmed system expense `kind = TAX`.

Frontend:

- Добавить `Настройки -> Налоги`.
- На выдаче заказа добавить radio только при включенной самозанятости.
- В summary выдачи показать сумму налога и итоговую чистую сумму после налога как справочную величину.

Docs:

- Добавить distant roadmap item: интеграция с `Мой налог`, без реализации.

### 4. Пользователи, мастера и комиссии

DB:

- `User`, `Membership`, `commissionPercent`, `isActive` уже есть.
- `RepairOrderItem.assignedMasterMembershipId`, `commissionPercentSnapshot`, `commissionBaseCents`, `commissionAmountCents`, `commissionCalculatedAt` добавлены для item-level commission.
- Возможно добавить `User.phoneNormalized`, если телефон пользователя станет поисковым полем.

API:

- Новый settings module:
  - `GET /api/v1/settings/members`;
  - `POST /api/v1/settings/members`;
  - `PATCH /api/v1/settings/members/:id`;
  - `DELETE /api/v1/settings/members/:id` или `PATCH isActive=false`.
- RBAC:
  - OWNER can manage admins/managers/masters except last owner protection;
  - ADMIN can create/update/deactivate MASTER only;
  - ADMIN cannot deactivate ADMIN/OWNER;
  - MASTER/MANAGER forbidden.

Frontend:

- `Настройки -> Мастера`.
- Список мастеров с active state и commission percent.
- Форма добавления мастера.
- Редактирование имени/email/phone/commission for masters.
- Disable/delete action only for allowed targets.
- В заказе выбирать мастера на каждой `SERVICE`-позиции.

Tests:

- admin cannot deactivate admin;
- admin can add/deactivate master;
- commission percent validates 0..100 and persists.
- issue calculates commission only from clean service-line base.
- admin/owner can mark service-line commission as paid.

### 5. Timezone в аналитике

Backend:

- Заменить UTC-only `monthPeriod()` на период в timezone организации.
- Использовать `auth.organization.timezone`.
- Переводить локальные границы месяца в UTC перед Prisma query.

Tests:

- Для `Asia/Yekaterinburg` начало месяца должно быть `previous UTC day 19:00:00.000Z` при UTC+5.
- KPI uses same local-month boundaries for orders, paidAt and expenses.

### 6. Audit module

DB migration:

- При необходимости добавить `ISSUE` в `AuditAction`.
- Остальные действия можно покрыть существующими `CREATE`, `UPDATE`, `CONFIRM`, `STATUS_CHANGE`, `PAYMENT_ADDED`.

Backend:

- Добавить shared audit helper.
- Писать audit после успешного бизнес-действия как best-effort, чтобы сбой журнала не ломал уже выполненную операцию.
- Новый API:
  - `GET /api/v1/audit?entityType=&entityId=&limit=`;
  - `GET /api/v1/repair-orders/:id/audit`.

Frontend:

- `Настройки -> Журнал` с последними событиями.
- Timeline на detail order.
- Фильтры по entity/action в общем журнале — в плане.

Tests:

- status change writes audit row;
- mark paid writes audit row;
- issue writes audit row;
- expense confirmation writes audit row;
- catalog create/update writes audit row;
- audit list is scoped by organization.

### 7. Customer profile editing

- Доменное решение:
  - имя, телефон, email и заметка живут в `Customer`;
  - старые заказы показывают актуальный профиль клиента;
  - before/after изменений хранится в `AuditLog`;
  - order-level snapshots и `CustomerRevision` оставить на будущую юридически строгую историю.
- Реализован `PATCH /api/v1/customers/:id`.
- Форма редактирования клиента добавлена в карточку заказа.

## Рекомендуемая последовательность

1. Расходы и номер заказа: самый заметный UX выигрыш без тяжелой модели.
2. Timezone analytics: небольшой backend change с понятными тестами.
3. Settings tax + issue radio: требует миграции, но ограничен одним flow.
4. Settings members: отдельный вертикальный slice API + UI + RBAC tests.
5. Audit writes + audit views: лучше делать после tax/users, чтобы логировать новые действия сразу.
6. Customer history design и редактирование профиля клиента: реализовано в текущей простой модели.

## Acceptance Criteria

- Общий расход создается из реестра расходов только после включения checkbox `Общий расход`.
- По умолчанию новый расход без контекста заказа предлагает поиск заказа, включая legacy формат `R-0001`.
- Расход к заказу или конкретной услуге создается из карточки заказа.
- Номер заказа в UI читается без обязательного `R-`.
- Поиск принимает старый и новый формат номера.
- В настройках можно включить самозанятость.
- При выдаче заказа можно выбрать физлицо или юрлицо/ИП.
- Налоговый snapshot сохраняется на заказе.
- Налог считается от текущего итога заказа, рассчитанного из позиций, независимо от расходов.
- Ручной расход с типом `TAX` создать нельзя.
- Комиссия мастера считается на уровне `SERVICE`-позиции после доли налога, себестоимости и подтвержденных расходников этой позиции, а не всего заказа.
- Выплата мастеру видна в реестре комиссий и может быть отмечена Owner/Admin.
- Admin может добавлять/отключать мастеров и менять комиссию.
- Admin не может отключить Admin/Owner.
- Аналитика использует timezone организации для границ периода.
- Важные действия пишутся в audit log.
- Audit виден в настройках и в timeline карточки заказа.
- Модель истории клиента зафиксирована.
- Клиента можно редактировать из карточки заказа, изменение пишется в audit log.
- Интеграция с `Мой налог` остается только в distant roadmap.
