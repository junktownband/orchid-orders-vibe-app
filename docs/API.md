# API

Base URL in local development: `http://localhost:3005`.

Protected endpoints require:

```http
Authorization: Bearer <accessToken>
```

Errors use the shared shape from `packages/shared/src/schemas/api.ts`:

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Validation error",
    "details": [],
    "errors": [
      {
        "field": "customer.phone",
        "message": "Invalid phone number"
      }
    ]
  }
}
```

## Auth

`POST /api/v1/auth/login`

```json
{
  "email": "sasha@orchid.local",
  "password": "orchid12345"
}
```

Returns `accessToken` and `user`, and sets the `orchid_refresh` httpOnly cookie.

`POST /api/v1/auth/refresh`

Uses the refresh cookie and returns a fresh `accessToken`, `user`, and rotated refresh cookie.

`POST /api/v1/auth/logout`

Clears the refresh cookie.

`GET /api/v1/auth/me`

Returns the current authenticated user context.

## Payments And Prepayments

Repair order payments are explicit backend records. Back-office users can add a prepayment from the order card before issue. The payment amount cannot be greater than the current order balance.

At issue time the backend does not accept a manually entered final amount. The order total is calculated from current order items. If prepayments do not cover the total, issue creates one payment for the remaining balance using the selected payment method.

The order response exposes `paidAmountCents`, `balanceDueCents`, and `payments[]`. `paymentStatus` is derived from total accepted payments:

- `UNPAID`: no accepted payment;
- `PARTIALLY_PAID`: accepted payments are lower than the order total;
- `PAID`: accepted payments exactly cover the order total;
- `VOIDED`: reserved for cancelled/voided payment semantics.

Payment correction is done through a void flow. Voided payments stay in history for audit/correction purposes but are excluded from accepted payment totals, order payment status recalculation, and dashboard paid revenue.

## Repair Orders

`GET /api/v1/repair-orders`

Returns a cursor-paginated repair order list scoped to the current user's organization. Order reads are transparent inside the organization, so every active role can read the list and detail endpoints.

Query parameters:

- `q` searches by customer name, customer phone, instrument brand/model/type, order number, and responsible master.
- `tab` can be `all`, `ready`, `active`, or `completed`.
- `repairStatus` can be `ACCEPTED`, `IN_PROGRESS`, `READY`, `ISSUED`, or `CANCELLED`.
- `paymentStatus` can be `UNPAID`, `PARTIALLY_PAID`, `PAID`, or `VOIDED`.
- `limit` defaults to `20` and is capped at `50`.
- `cursor` is the opaque `nextCursor` returned by the previous page.

Sorting is backend-owned: `READY` orders first, active orders next, issued/cancelled orders last; each group is ordered by latest update.

New repair order numbers are stored as zero-padded numeric strings such as `00001`. The UI displays them as `№ 00001`.
Search remains tolerant of legacy and human-entered formats: `00001`, `R-00001`, `R-0001`, `№ 00001`, and short numeric input can find the same order.

Search uses PostgreSQL trigram indexes on order number, customer name/phone, instrument fields, and master name.

```json
{
  "items": [],
  "nextCursor": null,
  "hasMore": false
}
```

`GET /api/v1/repair-orders/:id`

Returns one repair order card, including `customerPhone`, line items, statuses, assignee, payment summary, and payment history.

`GET /api/v1/repair-orders/:id/audit`

Returns the latest audit events scoped to the repair order. The web order card uses this endpoint for its timeline.

`POST /api/v1/repair-orders`

```json
{
  "customer": {
    "name": "Иван",
    "phone": "+7 900 000-00-00",
    "email": ""
  },
  "instrument": {
    "type": "guitar",
    "model": "Fender Stratocaster"
  },
  "description": "Отстройка",
  "assignedMasterMembershipId": "membership-id",
  "items": [
    {
      "name": "Отстройка",
      "type": "SERVICE",
      "assignedMasterMembershipId": "master-membership-id",
      "priceCents": 200000,
      "costCents": 0
    },
    {
      "name": "Комплект струн",
      "type": "STRINGS",
      "priceCents": 100000,
      "costCents": 60000
    }
  ]
}
```

`customer.phone` is optional. The API stores the submitted display value and maintains a normalized digit-only value for phone search.

`orderNumber` is generated by the backend and is not accepted in the create payload.

`GET /api/v1/repair-orders/masters`

Returns active organization members that can be assigned as the order coordinator or as a concrete service-line master.

`PATCH /api/v1/repair-orders/:id/assignee`

```json
{
  "assignedMasterMembershipId": "membership-id"
}
```

Use `null` or omit the value to clear the assignee.

`PUT /api/v1/repair-orders/:id/items`

Replaces editable order items before the order is issued.

```json
{
  "items": [
    {
      "id": "existing-item-id",
      "name": "Отстройка",
      "type": "SERVICE",
      "assignedMasterMembershipId": "master-membership-id",
      "priceCents": 200000,
      "costCents": 0
    }
  ]
}
```

`PATCH /api/v1/repair-orders/:id/status`

```json
{
  "repairStatus": "READY"
}
```

Allowed repair statuses: `ACCEPTED`, `IN_PROGRESS`, `READY`, `CANCELLED`. `ISSUED` is reached through the issue endpoint and locks the order. `OWNER`, `ADMIN`, and `MANAGER` can set normal repair statuses before issue; `MASTER` can only set `IN_PROGRESS` and `READY`.

`POST /api/v1/repair-orders/:id/mark-paid`

Adds a payment or prepayment to a non-issued order.

```json
{
  "amountCents": 500000,
  "paymentMethodId": "payment-method-id",
  "comment": "Предоплата"
}
```

`paymentMethodId` is required and must point to an active payment method. `amountCents` is optional; if omitted, the backend accepts the full current balance. If `amountCents` is lower than the balance, the order becomes `PARTIALLY_PAID`. If it equals the balance, the order becomes `PAID`. Values greater than the balance are rejected.

`POST /api/v1/repair-orders/:id/payments/:paymentId/void`

Voids an accepted payment and recalculates the order payment status from the remaining non-voided payments. Available to `OWNER` and `ADMIN`.

```json
{
  "reason": "Wrong payment method selected"
}
```

`reason` is required and is stored with the voided payment and audit event.

`POST /api/v1/repair-orders/:id/issue`

```json
{
  "paymentMethodId": "payment-method-id",
  "taxSubject": "INDIVIDUAL"
}
```

Issues the order using the current order total calculated from items. If the order has an unpaid balance, `paymentMethodId` is required and issue creates a payment for that remainder. If prepayments already cover the order total, `paymentMethodId` may be omitted. The order then becomes `ISSUED`, payment becomes `PAID`, and further edits are locked.

If organization tax mode is `SELF_EMPLOYED`, `taxSubject` is required:

- `INDIVIDUAL`: 4% from income from individuals.
- `BUSINESS`: 6% from income from legal entities and individual entrepreneurs.

Issue also stores order-level tax snapshots and creates one confirmed system `TAX` expense for the tax amount. Manual expense creation cannot set `kind = TAX`.

For commissions, issue stores service-line snapshots on each `SERVICE` item:

- `commissionPercentSnapshot`;
- `commissionBaseCents`;
- `commissionAmountCents`;
- `commissionCalculatedAt`.

The commission base is the service-line final revenue allocation minus the line's proportional order-tax share, line cost, and confirmed regular expenses linked to that line. Non-service/material lines do not generate master commission.

Confirmed regular item-level expenses are charged to the linked line/order total first, so the client-facing order total includes the consumable. The same expense is still treated as service cost and therefore remains in the commission base subtraction.

## Master Commissions

`GET /api/v1/commissions`

Returns service-line commission accruals for the current organization.

Optional query parameters:

- `masterMembershipId`: filter by concrete master membership;
- `payoutStatus`: `UNPAID` or `PAID`;
- `from`: inclusive accrual date in `YYYY-MM-DD`;
- `to`: inclusive accrual date in `YYYY-MM-DD`.

The period filters use `commissionCalculatedAt`, so the UI presents them as the commission accrual period.

Each item includes:

- repair order number and item name;
- assigned master;
- commission base, percent snapshot, and amount;
- payout status: `UNPAID` or `PAID`;
- payout timestamp and paying admin, if already paid.

`POST /api/v1/commissions/:repairOrderItemId/mark-paid`

Marks a service-line commission as paid to the master and creates one confirmed system expense with `kind = SALARY`.
The salary expense is linked to the repair order and concrete service line. Manual expense creation cannot set `kind = SALARY`.

`POST /api/v1/commissions/bulk-mark-paid`

Marks multiple service-line commissions as paid in one transaction and creates one confirmed system `SALARY` expense for each commission.

```json
{
  "repairOrderItemIds": ["item-id-1", "item-id-2"]
}
```

The request accepts 1 to 1000 item ids. If any selected commission is missing, already paid, or not payable, the transaction fails without partial payout side effects.

## Settings

`GET /api/v1/settings`

Returns organization settings, including tax mode and self-employed reference rates.

`PATCH /api/v1/settings/tax`

Available only to `OWNER`. `ADMIN` and `MANAGER` can read organization settings but cannot enable or disable tax mode.

Allowed values: `NONE`, `SELF_EMPLOYED`.

`GET /api/v1/settings/expense-categories`

Returns expense categories used by the expense form and list badges. `OWNER`, `ADMIN`, and `MANAGER` can read; `MANAGER` receives active rows only.

`POST /api/v1/settings/expense-categories`

Available to `OWNER` and `ADMIN`.

```json
{
  "name": "Материалы",
  "color": "#22c55e",
  "sortOrder": 10
}
```

`PATCH /api/v1/settings/expense-categories/:id`

Available to `OWNER` and `ADMIN`. Supports `name`, `color`, `isActive`, and `sortOrder`.

`GET /api/v1/settings/members`

Returns master accounts for the current organization. Available to `OWNER` and `ADMIN`.

`POST /api/v1/settings/members`

Creates or reactivates a master account. The API intentionally creates `MASTER` memberships only; admins cannot create or edit other admins from this screen.

```json
{
  "name": "Алексей",
  "email": "alex.master@orchid.local",
  "phone": "+7 900 000-00-00",
  "commissionPercent": 30
}
```

`commissionPercent` is user-facing percent in range `0..100`; the backend stores it as a decimal rate for commission snapshots.

`PATCH /api/v1/settings/members/:id`

Updates a master profile, phone, email, active state, or commission percent.

`DELETE /api/v1/settings/members/:id`

Soft-deactivates a master membership. Historical orders and commission snapshots keep their existing links. Admins cannot delete or downgrade admins/owners through this endpoint because it only targets `MASTER` memberships.

## Service Catalog

`GET /api/v1/service-catalog`

Returns non-deleted service catalog items scoped to the current user's organization, including inactive rows. Available to `OWNER`, `ADMIN`, and `MANAGER`.

`POST /api/v1/service-catalog`

The current shared schema accepts catalog creation for fixed services only.

```json
{
  "name": "Полная проточка ладов",
  "type": "SERVICE",
  "defaultPriceCents": 600000,
  "defaultCostCents": 50000
}
```

Catalog price and cost defaults are copied into order items. The copied cost reduces the service commission base immediately. Extra item-level expenses can still be added later for unusual materials or purchases.

`PATCH /api/v1/service-catalog/:id`

Updates a fixed service catalog item. The endpoint accepts any subset of editable fields and can deactivate/reactivate the row with `isActive`.

```json
{
  "name": "Full fret polish",
  "defaultPriceCents": 600000,
  "defaultCostCents": 50000,
  "isActive": false
}
```

Catalog edits and deactivation create before/after audit entries. Inactive services stay visible in service settings but are hidden from order service pickers.

## Expenses

`GET /api/v1/expenses`

Returns organization expenses.

`POST /api/v1/expenses`

```json
{
  "description": "Закупка струн",
  "categoryId": "expense-category-id",
  "paymentMethodId": "payment-method-id",
  "amountCents": 60000,
  "repairOrderId": "repair-order-id",
  "repairOrderItemId": "repair-order-item-id"
}
```

Expenses are created as `DRAFT`.

Manual expenses are always regular expenses. The public create schema rejects a submitted `kind` field, so `TAX` and `SALARY` can only be created by system flows such as repair order issue and master commission payout.

In the web UI, the expense form starts with order search rather than an order dropdown. Users can find an order by `00001`, `R-0001`, customer, phone, instrument, or master, or explicitly tick the general-expense checkbox to create an expense without `repairOrderId`. Order-linked expenses can also be started from the repair order card, where the order context is prefilled.

`POST /api/v1/expenses/:id/confirm`

Moves an expense to `CONFIRMED`. Only confirmed expenses are included in dashboard cash analytics.

If the expense is a regular expense linked to a concrete order item, confirmation also increases that item price and the order total by the expense amount. This charges the consumable/material to the client while still treating the confirmed expense as service cost for commission math.

`POST /api/v1/expenses/:id/void`

Voids a regular expense with a required reason. Voided expenses stay in history, are excluded from dashboard cash analytics, and no longer reduce unpaid service commission bases.

If the voided expense had charged a concrete order item, the backend reverses the item/order charge and recalculates unpaid commissions. The void is rejected if the adjusted order total would become lower than already accepted payments. System expenses such as `TAX` and `SALARY` cannot be manually voided.

```json
{
  "reason": "Duplicate receipt"
}
```

## Audit

`GET /api/v1/audit`

Returns the latest audit events for the current organization. Available to `OWNER`, `ADMIN`, and `MANAGER`.

Query parameters:

- `entityType` optionally filters by entity type, for example `RepairOrder`, `Expense`, or `ServiceCatalogItem`.
- `entityId` optionally filters by concrete entity id.
- `scope=finance` narrows the log to finance-related entities for the money section.
- `limit` defaults to `50` and is capped at `100`.

Tracked events currently include repair order creation, repair order item price/cost changes, assignee changes, status changes, payment marks, payment voids, order issue, expense creation, expense confirmations, expense voids, service catalog item creation/update/deactivation, tax setting changes, customer edits, master membership changes, and master commission payout marks. Audit writes are best-effort and must not break the original business action if logging fails after the action succeeds.

The web money section renders this endpoint as `История изменений` with human-readable rows. The API still returns `beforeJson`/`afterJson` for auditability, but the UI should not expose raw JSON as the primary presentation.

```json
{
  "items": [
    {
      "id": "audit-log-id",
      "entityType": "RepairOrder",
      "entityId": "repair-order-id",
      "action": "STATUS_CHANGE",
      "userName": "Owner",
      "beforeJson": null,
      "afterJson": {
        "repairStatus": "READY"
      },
      "comment": "Repair order status changed to READY",
      "createdAt": "2026-06-03T10:00:00.000Z"
    }
  ]
}
```

## Analytics

`GET /api/v1/analytics/dashboard`

Returns current-month KPI calculated on the backend.

Current product decision: `paidRevenueCents` is calculated from non-voided accepted payment records created inside the period. It is not calculated from full repair order totals by `paidAt`.

The default period is the current calendar month in the organization's timezone. The backend converts local month boundaries to UTC before querying PostgreSQL. For example, `Asia/Yekaterinburg` June 2026 starts at `2026-05-31T19:00:00.000Z`.

Commission fields:

- `accruedCommissionsCents`: all master commissions accrued in the period by `commissionCalculatedAt`;
- `paidCommissionsCents`: commissions marked as paid to masters within the dashboard period by `commissionPaidAt`;
- `payableCommissionsCents`: accrued commissions still unpaid;
- `netCashCents`: paid revenue minus confirmed expenses minus unpaid commissions. Confirmed expenses already include paid `SALARY` commission payouts, so paid commissions are not subtracted twice.
